## 开发策略

分 **3 个 Wave** 执行，每个 Wave 完成后可交付可用版本：

| Wave | 目标 | 预计工作量 | 对应角色 |
|:----:|------|:----------:|---------|
| **Wave 0** | 补课：修复已有需求但未实现的 P0 缺陷 | ~10-12 个日历日（1 前端 + 1 后端 + 1 数据并行） | 全角色 |
| **Wave 1** | 增强：P1 新需求 + 体验优化 | ~12-15 个日历日 | 全角色 |
| **Wave 2** | 高级：P2 自定义类别 + 时间线继承 | ~10-12 个日历日 | 按需 |

## 风险与缓解

| 风险 | 影响 | 缓解措施 |
|:----|:----:|:----------|
| Dashboard 1 秒刷新导致 UI 卡顿 | 用户体验下降 | 空闲/睡眠降频至 5-10 秒；轮询线程 BelowNormal；单次刷新耗时 > 200ms 时自动降至 2 秒 |
| **SQLite 写锁阻塞 Dashboard 查询** | 引擎每 30s 批量写入时锁住数据库，Dashboard 查询被阻塞导致刷新卡顿 | 启用 SQLite WAL 模式确保读写并发；Dashboard 查询使用只读连接；写入连接和读取连接分开 |
| **1s 刷新 + 分页 + 排序 + 搜索复合交互冲突** | 用户在搜索过滤后翻页，自动刷新导致数据重排，页面内容跳变 | 需求文档 F7.10 已明确交互策略：用户交互时暂停刷新，失焦 3 秒后恢复 |
| **Mutex 崩溃残留** | 程序崩溃后命名 Mutex 未释放，用户无法重新启动 | Mutex 获取失败时增加超时检测 + 检查对应进程是否存活；考虑使用 Mutex 的 AbandonedMutex 异常处理 |
| **系统关机时缓存数据丢失** | N1.3 每 30s 批量落盘，关机时最后一批数据可能丢失 | 监听 `SystemEvents.SessionEnding` 事件时强制刷盘；使用 SQLite `PRAGMA synchronous = FULL` 保证写入完整性 |
| 操作日志导出数据量过大 | 导出速度慢 | 仅导出当日数据；前台切换级别，非文件系统监听 |
| 自定义标签 + 类别继承数据模型复杂 | 开发延期 | Wave 2 优先做简单的一级标签，类别继承延后 |
| 测试覆盖不足 | 回归缺陷 | 每个任务完成时同步补充单元测试；总测试人力从 3.5 天提升至 12 天（占比 19%） |
