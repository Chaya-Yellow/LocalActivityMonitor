## 二、公共契约（Wave 0 · 必须先完成）

所有开发成员需要先就以下契约达成一致，之后各角色可独立开发。

### M1 数据模型定义

**文件**：`ActivityMonitor.Core/Models/`

| 类 | 字段 | 说明 |
|----|------|------|
| `ActivityEvent` | Id, StartTime, EndTime, DurationMs, Category, WorkTag, SubCategory, WindowTitle, ProcessName, ProcessPath, ProcessId, Detail, Domain, Project, Keywords, IsContinued, IsPrivate, IsCrashRecovered, EditedTitle, EditedDesc, UserCategory | 核心事件 POCO |
| `DailySummary` | Date, TotalActiveMs, TotalIdleMs, TotalSleepMs, AppBreakdown(JSON), DomainBreakdown(JSON), ProjectBreakdown(JSON), WorkMs, BreakMs, KeywordCloud(JSON), RawReport, UserNotes | 每日聚合 |
| `WeeklySummary` | WeekStart, WeekEnd, TotalActiveMs, TotalIdleMs, AppBreakdown(JSON), DomainBreakdown(JSON), ProjectBreakdown(JSON), AvgDailyHours | 每周聚合 |
| `MonthlySummary` | Month, TotalActiveMs, TotalIdleMs, AppBreakdown(JSON), DomainBreakdown(JSON), ProjectBreakdown(JSON), AvgDailyHours | 每月聚合 |

**依赖**：无
**产出**：`ActivityMonitor.Core/Models/` 下所有 POCO 类

---

### M2 接口契约定义

**文件**：`ActivityMonitor.Core/Interfaces/`

| 接口 | 核心方法 | 使用者 |
|------|----------|--------|
| `IActivityTracker` | `Start()`, `Stop()`, `Pause()`, `Resume()`, `event OnActivityChanged` | ActivityEngine → WindowTracker |
| `IActivityRepository` | `InsertAsync()`, `InsertBatchAsync()`, `GetTodayEventsAsync()`, `GetByDateAsync()`, `UpdateAsync()`, `DeleteAsync()`, `GetDailyStatsAsync()` | 所有模块 |
| `IActivityCategorizer` | `Classify(ActivityEvent) → (Category, WorkTag)` | ActivityEngine |
| `IIdleDetector` | `IsIdle`, `IdleThreshold`, `event OnIdleStateChanged` | ActivityEngine |
| `ISleepDetector` | `IsSleeping`, `event OnSleepStateChanged` | ActivityEngine |
| `ITodayStatsService` | `GetByAppAsync()`, `GetByProjectAsync()`, `GetByDomainAsync()`, `GetByCategoryAsync()`, `GetByWorkTagAsync()` | Dashboard |
| `IReportExporter` | `ExportDailyAsync(DateTime date) → string md` | Dashboard → ReportEditor |
| `ISettingsRepository` | `GetAsync(key)`, `SetAsync(key, value)` | 所有模块 |

**依赖**：M1（使用 ActivityEvent 等 Model）
**产出**：`ActivityMonitor.Core/Interfaces/` 下所有接口

### M3 SQLite 数据库层

**文件**：`ActivityMonitor.Data/`

| 任务 | 说明 |
|------|------|
| `SqliteContext.cs` | 数据库初始化、连接管理、迁移（建表/加索引） |
| `ActivityEventRepository.cs` | 实现 IActivityRepository：CRUD + 今日查询 + 聚合查询 |
| `DailySummaryRepository.cs` | 每日聚合表的读写 |
| `WeeklySummaryRepository.cs` | 每周聚合表的读写 |
| `MonthlySummaryRepository.cs` | 每月聚合表的读写 |
| `SettingsRepository.cs` | 设置项的读写 |

**DDL 语句**：

```sql
CREATE TABLE activity_events (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    start_time      TEXT NOT NULL,
    end_time        TEXT,
    duration_ms     INTEGER DEFAULT 0,
    category        TEXT NOT NULL,
    work_tag        TEXT DEFAULT 'unknown',
    sub_category    TEXT,
    window_title    TEXT,
    process_name    TEXT,
    process_path    TEXT,
    process_id      INTEGER,
    detail          TEXT,
    domain          TEXT,
    project         TEXT,
    keywords        TEXT,
    is_continued    INTEGER DEFAULT 0,
    is_private      INTEGER DEFAULT 0,
    is_crash_recovered INTEGER DEFAULT 0,
    edited_title    TEXT,
    edited_desc     TEXT,
    user_category   TEXT
);
CREATE INDEX idx_events_date ON activity_events(start_time);
CREATE INDEX idx_events_cat  ON activity_events(category, start_time);
CREATE INDEX idx_events_proc ON activity_events(process_name, start_time);

-- 每日/周/月聚合表 + settings 表（略，见 01-架构设计.md 五、数据模型）
```

**依赖**：M1（Models）、M2（IActivityRepository）
**产出**：`ActivityMonitor.Data/` 完整数据层

### M4 Win32 API 声明

**文件**：`ActivityMonitor.Core/Win32/NativeMethods.cs`

| 函数 | DLL | 用途 |
|------|-----|------|
| `GetForegroundWindow()` | user32.dll | 获取前台窗口句柄 |
| `GetWindowThreadProcessId()` | user32.dll | 从窗口句柄获取进程 ID |
| `GetWindowText()` | user32.dll | 获取窗口标题 |
| `GetLastInputInfo()` | user32.dll | 获取上次用户输入时间 |
| `RegisterPowerSettingNotification()` | user32.dll | 电源事件监听 |
| `QueryWorkingSet()` | psapi.dll | PS 文件句柄查询（Phase 2） |

**依赖**：无
**产出**：`ActivityMonitor.Core/Win32/NativeMethods.cs`

## 三、Wave 1 并行任务

完成 Wave 0 后，以下 4 个角色可同时开始工作。

---

### 角色A：核心追踪引擎

> 依赖：M1, M2, M3, M4

#### T1 WindowTracker（通用前台窗口追踪器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/WindowTracker.cs` |
| **实现** | `IActivityTracker` |
| **核心逻辑** | 2s 定时器轮询 `GetForegroundWindow()` → `GetWindowText()` → `GetWindowThreadProcessId()`，与当前活动记录对比，相同则累加时长，不同则创建新记录 |
| **排除规则** | `IntPtr.Zero`、托盘弹窗等不纳入 |
| **测试用例** | 模拟前台窗口切换、同一窗口持续、无窗口返回 Zero 三种场景 |
| **产出** | 可独立运行的窗口监控模块，输出 `(ProcessName, WindowTitle, DurationMs)` |

#### T2 IdleDetector（空闲检测器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/IdleDetector.cs` |
| **实现** | `IIdleDetector` |
| **核心逻辑** | `GetLastInputInfo()` 计算上次输入时间差；阈值 = 15 分钟；超过阈值触发 `OnIdleStateChanged` 事件 |
| **性能** | 每 2s 调用一次，线程 BelowNormal |
| **测试用例** | 模拟输入（重置计时器）、长时间不输入（触发空闲）、恢复输入（触发恢复） |
| **产出** | 空闲检测模块，通过事件通知引擎暂停/恢复 |

#### T3 SleepDetector（睡眠检测器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/SleepDetector.cs` |
| **实现** | `ISleepDetector` |
| **核心逻辑** | 注册 `WM_POWERBROADCAST` 消息，监听 `PBT_APMSUSPEND` / `PBT_APMRESUMEAUTOMATIC` |
| **产出** | 电源事件监听模块，通过事件通知引擎 |

#### T4 CrashRecoveryService（崩溃恢复）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/CrashRecoveryService.cs` |
| **核心逻辑** | 启动时读取退出标记文件（正常退出标记 vs 无标记=崩溃）；若崩溃且距上次记录 < 5 分钟，自动补录空白记录并标记 `is_crash_recovered=1` |
| **产出** | 崩溃恢复模块，启动时自动执行检查 |

#### T5 ActivityEngine（主协调器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/ActivityEngine.cs` |
| **职责** | 组装上述 T1-T4，协调追踪生命周期 |
| **核心逻辑** | `Start()` → 启动 Timer，每 2s：检查 Idle/Sleep → WindowTracker 获取当前窗口 → Categorizer 分类 → Repository 写入 → 统计更新 |
| **产出** | 完整的追踪引擎，对外暴露 `Start/Pause/Resume/Stop` |

### 角色B：内容解析与分类

> 依赖：M1, M2, M3, M4

#### C1 BrowserTracker（浏览器追踪器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/BrowserTracker.cs` |
| **核心逻辑** | 从窗口标题解析页面标题和域名：`"标题 - Google Chrome"` → 截去 suffix 得到标题，从标题中提取域名规则（URL 关键词匹配） |
| **支持列表** | Chrome/Edge/Firefox 的标题格式 |
| **测试用例** | `"GitHub - Google Chrome"` → `(title="GitHub", domain="github.com")` |
| **产出** | 浏览器解析模块，输出 `(Title, Domain)` |

#### C2 FileTracker（文件追踪器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/FileTracker.cs` |
| **核心逻辑** | 根据窗口标题和进程名称推断操作的文件名：VS Code 标题提取 `文件名 + 项目名`；PS 通过进程工作目录 + 文件句柄推断；终端读取标题中的路径；远程桌面记录目标地址 |
| **已知应用解析表** | 见 `01-架构设计.md` 4.3 节 |
| **测试用例** | `"app.py - myproject - Visual Studio Code"` → `(file="app.py", project="myproject")` |
| **产出** | 文件解析模块，输出 `(FileName, FilePath, ProjectName)` |

#### C3 ProjectDetector（项目检测器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Classification/ProjectDetector.cs` |
| **核心逻辑** | 从文件路径或进程路径中检测项目信息：向上遍历寻找 `.git` 目录；取最后两级文件夹名作为项目名；兼容 VS Code / 终端 / PS 三种场景 |
| **测试用例** | `C:\Users\name\Code\MyProject\src\main.py` → `MyProject` |
| **产出** | 项目检测模块，输出项目名 |

#### C4 ActivityCategorizer（活动分类器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Classification/ActivityCategorizer.cs` |
| **实现** | `IActivityCategorizer` |
| **核心逻辑** | 按规则表分类：进程名→类别（web/file/app/idle/sleep）；工作/非工作基于域名+关键词+用户历史标记 |
| **规则表** | 见 `01-架构设计.md` 4.7 节 |
| **测试用例** | `{ProcessName="chrome.exe", WindowTitle="bilibili"}` → `(Category.Web, WorkTag.Unknown)`（需用户确认） |
| **产出** | 分类器，输出 `(Category, WorkTag)` |

#### C5 KeywordExtractor（关键词提取）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Classification/KeywordExtractor.cs` |
| **核心逻辑** | 从窗口标题中去除停用词（的、了、是、在…）、标点符号，提取有意义的词组；支持中文分词基础支持 |
| **测试用例** | `"Linux 内核编译参数 - Google Chrome"` → `["Linux", "内核", "编译", "参数"]` |
| **产出** | 关键词提取模块 |

#### C6 TodayStatsService（当日实时统计）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Classification/TodayStatsService.cs` |
| **实现** | `ITodayStatsService` |
| **核心逻辑** | 查询当天 `activity_events`（00:00 至今），按 process_name / project / domain / work_tag 分组聚合 `SUM(duration_ms)`，返回占比数据 |
| **聚合维度** | 按软件、按项目、按域名、按类别、按工作/非工作 |
| **产出** | 实时统计数据，供 Dashboard 展示 |

## 五、角色分配建议

| 角色 | 负责模块 | 预估工作量 | 前置条件 |
|------|----------|-----------|----------|
| **角色A** | 追踪引擎（T1-T5） | ~3 天 | 等待 M1-M4 完成 |
| **角色B** | 内容解析与分类（C1-C6） | ~2 天 | 等待 M1-M4 完成 |

## 六、代码分支策略

```
main ← 稳定版，只合入经过验证的代码
├── feature/engine       ← 角色A
├── feature/classification  ← 角色B
├── feature/ui           ← 角色C
├── feature/report       ← 角色D
└── feature/core         ← Wave 0（M1-M4，完成后合并到 main）
```

每个 feature 分支开发完成后 → PR 合入 `main`。
