## 二、Wave 0 — 技术架构师职责

> **负责人：技术架构师**
> Wave 0 定义整个项目的基石，所有后续角色的工作都依赖于这里产出的契约。
> 架构师需要输出完整的接口定义、数据模型、数据库 DDL 和 Win32 声明，
> 并在合并到 main 后给每个角色提供明确的接口文档。

### M1 数据模型定义

**文件**：`ActivityMonitor.Core/Models/`

| 类 | 字段 | 说明 |
|----|------|------|
| `ActivityEvent` | Id, StartTime, EndTime, DurationMs, Category, WorkTag, SubCategory, WindowTitle, ProcessName, ProcessPath, ProcessId, Detail, Domain, Project, Keywords, IsContinued, IsPrivate, IsCrashRecovered, EditedTitle, EditedDesc, UserCategory | 核心事件 POCO |
| `DailySummary` | Date, TotalActiveMs, TotalIdleMs, TotalSleepMs, AppBreakdown(JSON), DomainBreakdown(JSON), ProjectBreakdown(JSON), WorkMs, BreakMs, KeywordCloud(JSON), RawReport, UserNotes | 每日聚合 |

**依赖**：无
**产出**：`ActivityMonitor.Core/Models/` 下所有 POCO 类

### M3 SQLite 数据库层

**文件**：`ActivityMonitor.Data/`

| 任务 | 说明 |
|------|------|

| `SettingsRepository.cs` | 设置项的读写 |

**DDL 语句**：

```sql
CREATE TABLE activity_events (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    start_time      TEXT NOT NULL,
    end_time        TEXT,
    duration_ms     INTEGER DEFAULT 0,
    category        TEXT NOT NULL,
    work_tag        TEXT DEFAULT 'unknown',
    sub_category    TEXT,
    window_title    TEXT,
    process_name    TEXT,
    process_path    TEXT,
    process_id      INTEGER,
    detail          TEXT,
    domain          TEXT,
    project         TEXT,
    keywords        TEXT,
    is_continued    INTEGER DEFAULT 0,
    is_private      INTEGER DEFAULT 0,
    is_crash_recovered INTEGER DEFAULT 0,
    edited_title    TEXT,
    edited_desc     TEXT,
    user_category   TEXT
);
CREATE INDEX idx_events_date ON activity_events(start_time);
CREATE INDEX idx_events_cat  ON activity_events(category, start_time);
CREATE INDEX idx_events_proc ON activity_events(process_name, start_time);

-- 每日/周/月聚合表 + settings 表（略，见 01-架构设计.md 五、数据模型）
```

**依赖**：M1（Models）、M2（IActivityRepository）
**产出**：`ActivityMonitor.Data/` 完整数据层

## 三、Wave 1 并行任务

完成 Wave 0 后，以下 5 个角色可同时开始工作。

---

### 角色A：后端引擎工程师

> **角色定位**：负责整个追踪引擎的运行时核心，确保 2s 周期稳定运行、空闲/睡眠/崩溃场景全部覆盖。
> **依赖**：M1, M2, M3, M4

#### T1 WindowTracker（通用前台窗口追踪器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/WindowTracker.cs` |
| **实现** | `IActivityTracker` |
| **核心逻辑** | 2s 定时器轮询 `GetForegroundWindow()` → `GetWindowText()` → `GetWindowThreadProcessId()`，与当前活动记录对比，相同则累加时长，不同则创建新记录 |
| **排除规则** | `IntPtr.Zero`、托盘弹窗等不纳入 |
| **测试配合** | 角色E → E2-1 Mock 窗口句柄，验证状态机变换 |
| **产出** | 可独立运行的窗口监控模块，输出 `(ProcessName, WindowTitle, DurationMs)` |

#### T2 IdleDetector（空闲检测器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/IdleDetector.cs` |
| **实现** | `IIdleDetector` |
| **核心逻辑** | `GetLastInputInfo()` 计算上次输入时间差；阈值 = 15 分钟；超过阈值触发 `OnIdleStateChanged` 事件 |
| **性能** | 每 2s 调用一次，线程 BelowNormal |
| **测试配合** | 角色E → E2-2 模拟 LASTINPUTINFO 变化，验证状态切换 |
| **产出** | 空闲检测模块，通过事件通知引擎暂停/恢复 |

#### T3 SleepDetector（睡眠检测器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/SleepDetector.cs` |
| **实现** | `ISleepDetector` |
| **核心逻辑** | 注册 `WM_POWERBROADCAST` 消息，监听 `PBT_APMSUSPEND` / `PBT_APMRESUMEAUTOMATIC` |
| **测试配合** | 角色E → E2-3 模拟电源事件消息，验证自动暂停/恢复 |
| **产出** | 电源事件监听模块，通过事件通知引擎 |

#### T4 CrashRecoveryService（崩溃恢复）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/CrashRecoveryService.cs` |
| **核心逻辑** | 启动时读取退出标记文件（正常退出标记 vs 无标记=崩溃）；若崩溃且距上次记录 < 5 分钟，自动补录空白记录并标记 `is_crash_recovered=1` |
| **测试配合** | 角色E → E2-4 模拟删除标记文件，验证补录逻辑 |
| **产出** | 崩溃恢复模块，启动时自动执行检查 |

#### T5 ActivityEngine（主协调器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/ActivityEngine.cs` |
| **职责** | 组装上述 T1-T4，协调追踪生命周期 |
| **核心逻辑** | `Start()` → 启动 Timer，每 2s：检查 Idle/Sleep → WindowTracker 获取当前窗口 → Categorizer 分类 → Repository 写入 → 统计更新 |
| **测试配合** | 角色E → E2-5 集成测试：Mock 全部依赖，验证完整 tick 流程 |
| **产出** | 完整的追踪引擎，对外暴露 `Start/Pause/Resume/Stop` |

### 角色B：内容分析工程师

> **角色定位**：负责将原始窗口标题转化为有意义的结构化数据（文件名、域名、项目名、关键词、分类标签）。

| **文件** | `ActivityMonitor.Core/Tracking/BrowserTracker.cs` |
| **核心逻辑** | 从窗口标题解析页面标题和域名：`"标题 - Google Chrome"` → 截去 suffix 得到标题，从标题中提取域名规则（URL 关键词匹配） |

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Tracking/FileTracker.cs` |

#### C3 ProjectDetector（项目检测器）

| 项目 | 内容 |
|------|------|
| **文件** | `ActivityMonitor.Core/Classification/ProjectDetector.cs` |

| 项目 | 内容 |

| **实现** | `IActivityCategorizer` |

#### C5 KeywordExtractor（关键词提取）

| 项目 | 内容 |

#### C6 TodayStatsService（当日实时统计）

| 项目 | 内容 |
|------|------|

## 六、产品经理助手（PM Assistant）

**这不是一个开发角色，而是一个需求把关角色。**

### 定位

在 `doc/00-需求文档.md`（需求规格）之前，增加一道**需求审查关卡**。
每当产品经理提出新想法，先由 PM Assistant 审查，再进入开发流程。

### 审查框架

| 维度 | 审查内容 |
|------|----------|
| **完整性** | 谁、什么场景、要什么？验收标准是啥？ |
| **矛盾检查** | 是否和已有功能/历史决策/设计原则冲突？ |
| **边界情况** | 数据为空？异常？网络问题？权限不足？ |
| **影响评估** | 改哪些模块？要改数据模型吗？性能影响？ |
| **优先级建议** | P0/P1/P2？有没有更简单的替代方案？ |

### 工作流

```
产品经理 → 提出新需求
              │
              ▼
      启动 pm-assistant agent
              │
              ▼
      输出审查报告（5 个维度）
              │
              ▼
      产品经理决定：接受 / 修改 / 拒绝
              │
              ▼（确认后）
      更新需求文档 → 更新架构设计 → 拆分任务 → 开发
```

### 配置文件

定义在 `.claude/agents/pm-assistant.md`，只有读权限（Read + WebSearch + AskUserQuestion），
不写代码、不改文档。

## 七、代码分支策略

```
main ← 稳定版，只合入经过 review + 测试通过的代码
├── feature/core           ← Wave 0（M1-M4 · 架构师）
```

- 合入前必须通过全部 E1-E5 测试
