### FR-001：实时统计面板数据不会动态刷新

**用户原话：**
> 总活跃时长不会动态变更，只停留在打开的那一刻算总活跃时长，工作时长、空闲/休息，事件数也好像静止了，界面上突然打开某个窗口页面，事件数不会+1。

**涉及已有需求：** F7.6（Dashboard 定时刷新）、F7.1（动态统计 00:00 到当前时间）

#### ✅ 明确的部分
- 问题现象清楚：Dashboard 打开后数据不自动刷新，停留在打开瞬间的快照
- 涉及三个指标：总活跃时长、工作时长/空闲时长、事件计数
- 窗口切换后事件数未增加，说明新活动未被实时计入

#### ⚠️ 模糊/遗漏
- 是否手动刷新（F5 / 按钮）有效？还是连手动刷新也不更新？A：不需要手动刷新，需要自动刷新
- 是关闭 Dashboard 重新打开后数据变正常，还是一直卡住？A：重新打开也不更新
- 刷新频率期望是多少？1s / 2s / 5s？A：1s
- 时间线视图是否也跟着停滞了？A：是的

#### ❌ 潜在矛盾
- F7.6 明确要求"Dashboard 定时刷新以保证数据实时性"，但当前行为未实现该需求 → **这是一个 Bug 或未实现项，而非新需求**

#### 🔍 边界场景
- 电脑从睡眠恢复后，Dashboard 是否重新开始刷新？A：需要
- 长时间挂机（隔夜）后 Dashboard 是否还能恢复刷新？A：需要
- 如果轮询刷新导致 UI 卡顿，是否需要降频策略？A：需要

#### 💡 补充挖掘
用户提到的"事件数"可能是指活动事件总条数（`activity_events` 行数）。结合 FR-003 的日志需求，用户对"每一个操作是否被记录"有很强的关注。建议在 Dashboard 上增加一个小型状态指示器（如"最后刷新：几秒前"），让用户感知到系统在实时工作。

#### 📊 建议优先级
**P0 — 核心阻断**。实时统计是 Dashboard 的首要功能，数据不刷新则面板毫无意义。需优先排查是定时器未启动、数据源未更新还是 UI 绑定问题。

### FR-002：浏览器追踪数据准确性质疑

**用户原话：**
> 今天下午4点半-5点，浏览器没有任何记录显示有打开过 taobao.com，你这个数据是哪里来的。

**涉及已有需求：** F2（浏览器活动追踪）、F2.2（支持 Chrome/Edge/Firefox 标题解析）

#### ✅ 明确的部分
- 用户自证在该时段未打开过 taobao.com
- 怀疑系统产生了虚假记录（误报/误解析）

#### ⚠️ 模糊/遗漏
- 用户是通过什么方式确认"浏览器没有任何记录"？（浏览器历史？还是凭记忆？）A：浏览器历史无该记录
- 那条 taobao.com 记录的具体信息是什么？（进程名、窗口标题、起始/结束时间、域名解析方式）A：显示购物信息-Chrome，下方显示taobao.com
- 是 Chrome 的标题被误解析为 taobao.com，还是完全另一个进程？A：不太清楚软件是怎么记录这个地址的。
- 是否可能是广告弹窗/后台页面/浏览器预加载导致的？A：不排除这种情况，但是浏览器已经禁用广告，所以没有看到广告弹出来。

#### ❌ 潜在矛盾
- 系统只追踪前台窗口，如果用户当时确实没有操作 taobao.com，可能是：① 其他软件的窗口标题包含 taobao 字符；② 浏览器预加载；③ 标题解析算法将某个域名误匹配为 taobao.com

#### 🔍 边界场景
- 浏览器同时开多个标签页，标题只显示当前激活的标签 — 但域名解析只取窗口标题中的内容
- 浏览器扩展/插件自动打开后台标签页（如淘宝比价插件）
- 淘宝的推广弹窗（即使通过其他网站打开）
- 恶意软件/浏览器劫持
- 用户可能开了隐私/无痕窗口但忘记

#### 💡 补充挖掘
这条反馈背后是**信任危机**。一旦用户认为数据不可靠，整个产品的价值就动摇了。建议：
1. **增加"来源追溯"**：每条活动记录右键查看原始窗口标题+进程路径，让用户能核实
2. **域名解析加置信度标记**：如 `taobao.com (从标题 "淘宝 - Chrome" 解析，置信度: 高)` 或如果是从标题中间匹配的显示 `(模糊匹配)`
3. **可行性评估**：考虑数据修正功能，用户可以标记某条记录为"误报"

#### 📊 建议优先级
**P0 — 核心阻断**。数据准确性是产品的生命线。不解决信任问题，其他功能再好也没用。

### FR-003：需要操作详情日志

**用户原话：**
> 需要有个详情日志，例如哪一分哪一秒对某个文件进行了什么操作。

**涉及已有需求：** F1.1（2 秒轮询记录前台窗口）、F3（文件与应用活动追踪）

#### ✅ 明确的部分
- 用户想要比当前"窗口切换"更细粒度的操作日志
- 粒度要求到"分/秒"级别
- 操作对象明确：文件
- 操作类型未明确说明

#### ⚠️ 模糊/遗漏
- "什么操作"具体指什么？打开文件 / 编辑文件 / 保存 / 关闭 / 重命名 / 删除？A：是的，你说的所有操作，还包括最小化。
- 是所有文件都需要，还是只限通过特定软件（IDE、Office、PS）打开的文件？A：所有exe，资源管理器也算。但是后台一直在运行的就不算了。
- 日志的查看位置：Dashboard 上？独立窗口？还是导出到日报中？A：导出按钮-弹窗-预览日志内容-可编辑-确认导出
- 日志保留策略：和活动数据一样按保留策略走，还是单独配置？A：只导出当日就行
- 记录频率：每次操作都记录（可能量很大）还是只记录文件每次切换？A：文件切换。比如最小化的就不记录了，只记录显示在桌面的内容。

#### ❌ 潜在矛盾
- 当前架构只追踪前台窗口（2 秒轮询），不做文件系统级别的监听。要实现"对某个文件进行了什么操作"，需要文件系统监控（ReadDirectoryChangesW / FileSystemWatcher），这显著增加了系统资源消耗，与 N1.1（内存 < 50MB）和 N1.2（用户无感知）存在冲突
- N2.2 明确承诺"不录制屏幕、不捕获键盘内容"——文件级别的操作日志（如"修改了第几行"）可能越界

#### 🔍 边界场景
- 用户快速操作（每秒多次保存）—— 日志量暴增
- 文件在外部程序中被修改（记事本改了，然后切回 IDE）
- 网络文件/云盘文件的操作追踪
- 文件被删除后，日志如何展示？

#### 💡 补充挖掘
用户的真实需求可能没到"监控文件系统事件"那么重，更可能是：
1. **想知道某个时间段在哪个项目/文件上工作** → 这其实可以通过增强窗口标题解析实现（如 VS Code 标题会显示当前文件名）
2. **还原工作现场** → 结合 FR-008 的多项目并行记录，用户更像是在追踪"我什么时间做了什么"
3. **建议折中方案**：活动记录中增加 `window_title_snapshot` 字段存储每次轮询的完整标题，在详情中展示"时间线文本流"，让用户看到文件名的变化序列，但不增加文件系统监听

#### 📊 建议优先级
**P1 — 重要但需细化**。先搞清楚用户想要的是文件操作审计级别还是时间线还原级别，后者可以用现有数据以更低成本实现。

### FR-004：监控状态无法暂停和开始

**用户原话：**
> 监控状态无法暂停跟开始。

**涉及已有需求：** F11.3（托盘右键菜单：暂停/恢复监控）、Q9（暂停监控功能）、N2.4（用户可随时暂停监控）

#### ✅ 明确的部分
- 用户找不到或无法使用暂停/恢复功能

#### ⚠️ 模糊/遗漏
- 是托盘菜单中没有"暂停/恢复"选项（未实现）？A：托盘有，但是在Dashboard没有显示。
- 还是菜单中有但点击无效（Bug）？A：无该内容。
- 用户是在 Dashboard 中找的还是托盘菜单中找的？A：托盘。
- 暂停后计时器/UI 状态是否有反馈？A：无反馈。

#### ❌ 潜在矛盾
- **严重矛盾**：F11.3 和 N2.4 都明确要求了暂停/恢复功能，但如果用户反馈找不到，要么是功能未实现，要么是交互入口不够明显
- 回溯现有代码：如果托盘菜单只实现了"打开面板"和"退出"，但缺少"暂停/恢复"，则属于开发未完成

#### 🔍 边界场景
- 暂停后的事件如何处理？丢弃还是缓存，恢复后补录？A：丢弃，也就是不进行监控状态。

- 暂停时长是否要在日报中显示？A：需要。

#### 💡 补充挖掘
结合 FR-001（数据不刷新），暂停功能缺失可能是同一个原因的两种表现：**监控引擎的状态管理不完整**。建议：
1. 托盘菜单保证有"暂停监控"和"恢复监控"（根据当前状态动态切换文字）
2. 暂停后托盘图标改变颜色（绿→灰）
3. Dashboard 上显示一个大状态条："⚠️ 监控已暂停"
4. 建议增加快捷键（如托盘图标双击切换暂停/恢复）

#### 📊 建议优先级
**P0 — 核心阻断**。暂停监控是隐私承诺（N2.4）的重要组成部分，功能缺失意味着用户无法在隐私敏感场景下使用。

### FR-006：历史记录统计多维增强

**用户原话：**
> 历史记录统计要有日、周、月、半年、一年，日和周可以查看详情操作，要包含年月日时分秒软件；月、半年、一年则显示不同软件使用占比、不同文件夹下的使用占比、统计每日使用时长、每周平均下来的每日时长（包含相比于上周多或者少了多长时间）、每半年和一年同理。

**涉及已有需求：** F10（周报/月报）、F8.4（历史数据浏览）

#### ✅ 明确的部分
- 时间粒度扩展：日、周、月、半年、一年
- 日和周 → 可查看详情操作（含年月日时分秒、软件名）
- 月/半年/一年 → 统计视图：软件占比、文件夹占比、每日时长、周均每日时长
- 需要环比比较：相比上周多/少了多长时间

#### ⚠️ 模糊/遗漏
- "查看详情操作"的"操作"是指 FR-003 的详情日志，还是指活动事件明细列表？A：活动时间明细。
- 周/月/半年/一年的起始点如何定义？（自然周（周一~周日）？自定义？）A：自然周。
- 环比对比的基准：周同比是跟上个自然周比，还是跟上个同样天数的周期比？A：自然周。
- 半年和一年的对比基准是什么？（去年同半年？上个半年？）A：上个半年。
- "不同文件夹下的使用占比" — 这里的"文件夹"是指项目文件夹还是任意文件夹？A：任意。
- 数据视图的呈现形式（图表/表格/混合）？A：混合。
- 半年/一年的每日平均时长，是仅计算有数据的日期，还是按日历日全部计算？A：按日历。

#### ❌ 潜在矛盾
- 原始需求 F10 只要求到月级别（P2 优先级），用户现在要求到半年和一年级别
- 半年/一年的数据量可能很大，Dashboard 一次性加载可能影响性能（与 N1.1 冲突）
- 现有的 `MonthlySummary` 模型可能需要扩展或新增 `HalfYearlySummary` / `YearlySummary`
- 环比需要存储历史周期的聚合快照，或每次实时计算（性能权衡）

#### 🔍 边界场景
- 使用不足一周——周均每日时长怎么算？A：使用时长/7，保留小数点后2位。
- 使用不足半年——半年统计是否显示"数据不足"？A：使用时长/半年时间，保留小数点后2位。
- 跨年数据——一年统计是按自然年还是滚动365天？A：自然年。
- 中途有几天完全没有数据（休假/出差）——周均是否排除这些天？A：否，空的需要有地方进行批注。
- 数据保留期 30 天的情况下，半年/一年统计无法生成（矛盾）A：数据保留30天，那就最多可生成近30天的统计。

#### 💡 补充挖掘
这个需求非常丰富但也非常重。建议分阶段：
1. **Phase 1（P1）**：实现"日"和"周"级别的详细视图，复用现有的日报和周报逻辑
2. **Phase 2（P2）**：实现"月"级别的统计视图
3. **Phase 3（P3）**：实现"半年"和"一年"视图 + 环比对比

其中环比对比（相比上周多/少）是一个非常有价值的功能但实现复杂，需要决定是存储历史快照还是实时聚合。

#### 📊 建议优先级
**P1（基础日/周）→ P2（月）→ P3（半年/一年/环比）**。先保证日和周的可用性，再逐步扩展。

### FR-007：电脑锁屏算作休息时间

**用户原话：**
> 电脑锁屏算作休息时间。

**涉及已有需求：** F4.5（睡眠/休眠自动暂停，标记为"睡眠"）、F4.3（空闲时段显示时长）

#### ✅ 明确的部分
- 锁屏（Win+L）触发的状态应归类为"休息"，而非"空闲"或"睡眠"
- 需要在日报/统计中正确体现

#### ⚠️ 模糊/遗漏
- 锁屏后用户可能还在（只是暂时离开），锁屏期间算休息还是算空闲？A：锁屏算休息。
- 锁屏后系统可能进入睡眠，两者如何区分？A：统一算为休息。
- "休息时间"是否需要用户手动标记为"工作/非工作"中的"break"？A：自动填写上break。

#### ❌ 潜在矛盾
- 当前系统通过电源事件监听（`RegisterPowerSettingNotification`）检测睡眠/休眠，但锁屏（`LockWorkStation` / `Win+L`）不是一个系统电源事件，而是桌面锁定事件
- 需要额外的 Win32 API（`WTSRegisterSessionNotification` 或监听 `WM_WTSSESSION_CHANGE`）来检测锁屏/解锁
- F4.3 空闲时段用户已可以编辑补充描述（如"午休"），锁屏自动标记为"休息"比手动更智能

#### 🔍 边界场景
- 锁屏后很快解锁（去倒杯水 < 1 分钟）——是否算休息？
- 锁屏后进入睡眠——双重事件如何处理？优先算睡眠还是休息？
- 远程桌面锁定（RDP 断开）——是否算锁屏？

- 用户双击 exe 但程序正在启动中（启动较慢）——防多实例的互斥体（Mutex）检查时机

#### 💡 补充挖掘
单例模式在 C# / WPF 中通常通过 `Mutex` 实现：

```
// 使用命名 Mutex 检测已有实例
bool createdNew;
using (var mutex = new Mutex(true, "Local\\ActivityMonitor-{GUID}", out createdNew))
{
    if (!createdNew)
    {
        // 激活已有窗口，退出当前进程
        return;
    }
    Application.Run();
}
```

对于关闭按钮行为，建议实现方案：
1. 所有窗口统一拦截 `Window.Closing` 事件
2. 弹窗提供两个选项："🎯 加载到后台继续监控"和"❌ 彻底退出"
3. "加载到后台"后托盘图标保持运行，所有窗口关闭
4. 托盘右键菜单增加"退出"选项，点击同样弹确认框

#### 📊 建议优先级
**P0 — 核心阻断**。多实例会导致数据冲突（两个进程写同一个 SQLite 文件）、重复记录、UI 混乱。关闭按钮行为影响所有用户每次使用的基础体验。
