# 活动监控器 — 开发规范

> 创建日期：2026-07-20
> 适用于：ActivityMonitor 项目所有 C# .NET 8 WPF 代码

---

## 一、总则

1. **可读性优先** — 代码是写给人读的，其次才是给机器执行
2. **一致性** — 新代码必须匹配文件现有风格
3. **最小意外** — 遵循 .NET 社区和微软官方约定，不发明私有的命名/模式
4. **性能敏感** — 后台追踪引擎的性能路径（2s 轮询）需特别关注分配和延迟

---

## 二、命名规范

### 2.1 标识符大小写

| 元素 | 大小写 | 示例 |
|------|--------|------|
| 类 / 结构体 / 记录 | PascalCase | `WindowTracker`, `ActivityEvent` |
| 接口 | PascalCase + I 前缀 | `IActivityTracker`, `IRepository` |
| 方法 | PascalCase | `GetForegroundWindow()`, `SaveAsync()` |
| 属性 | PascalCase | `StartTime`, `ProcessName` |
| 公开字段 | PascalCase（尽量用属性替代） | 不直接暴露公开字段 |
| 私有字段 | _camelCase | `_timer`, `_currentEventId` |
| 私有静态字段 | s_camelCase | `s_defaultThreshold` |
| 局部变量 / 参数 | camelCase | `windowHandle`, `eventId` |
| 常量 | PascalCase | `DefaultIdleThresholdMinutes` |
| 枚举 | PascalCase（枚举值 = PascalCase） | `Category.Web`, `WorkTag.Work` |
| 命名空间 | PascalCase, 点分 | `ActivityMonitor.Core.Tracking` |

### 2.2 缩写

- 不超过 2 个大写字母的缩写全大写：`IO`, `DB`, `UI`, `API`
- 超过 2 个字母的缩写按单词首字母大写：`XmlParser`, `HtmlExporter`（而非 `XMLParser`, `HTMLExporter`）
- 特例：`Id` 而非 `ID`（`ProcessId` 而非 `ProcessID`）

### 2.3 文件与文件夹

| 元素 | 规范 |
|------|------|
| 文件名 | 与类名一致：`WindowTracker.cs` |
| 每个文件一个公开类型 | 除小型枚举/委托外，不允许多个公开类型 |
| 文件夹名 | PascalCase，与命名空间最后一段一致：`Tracking/`, `Classification/` |
| XAML 文件 | 与 Code-behind 类名一致：`DashboardWindow.xaml` / `DashboardWindow.xaml.cs` |

## 三、项目结构规范

### 3.1 分层依赖方向

```
TrayApp (UI)  →  Core (逻辑)  →  Data (持久化)
     ↓                   ↓
  引用 Core          引用 Data
```

- **Core** 层不依赖任何其他项目层，只依赖 .NET BCL
- **Data** 层可引用 Core（使用其 Model）
- **TrayApp** 引用 Core 和 Data
- 禁止循环依赖

### 3.2 命名空间映射

```
文件夹路径                           命名空间
src/ActivityMonitor.Core/Models/     ActivityMonitor.Core.Models
src/ActivityMonitor.Core/Tracking/   ActivityMonitor.Core.Tracking
src/ActivityMonitor.Data/Database/   ActivityMonitor.Data.Database
src/ActivityMonitor.TrayApp/Dashboard/  ActivityMonitor.TrayApp.Dashboard
```

### 3.3 文件组织原则

```
类文件：    WindowTracker.cs           ← 主要逻辑
分部类：    WindowTracker.Events.cs    ← 事件处理分部
接口：      IWindowTracker.cs          ← 同一文件夹
枚举：      Category.cs                ← 小型枚举内联在使用处附近
            大型枚举放 Models/Enums/ 文件夹
```

## 四、代码风格

### 4.1 格式化规则（EditorConfig）

始终保持以下规则（由 `.editorconfig` 文件强制执行）：

```
indent_style = space
indent_size = 4
csharp_new_line_before_open_brace = all
csharp_preferred_modifier_order = public, private, protected, internal, static, extern, new, virtual, abstract, sealed, override, readonly, unsafe, volatile, async
```

### 4.2 花括号

- **Allman 风格**：左花括号换行
- 即使单行也必须使用花括号（禁止省略 `if` / `for` 花括号）

```csharp
// ✅ 正确
if (condition)
{
    DoSomething();
}

// ❌ 错误
if (condition) DoSomething();
if (condition) { DoSomething(); }
```

### 4.3 using 语句

- `using` 指令放在文件顶部，按 系统 → NuGet → 项目 顺序分组
- 启用 Implicit Usings（.NET 8 默认）
- 文件顶部不包含 `global using`（放在 `GlobalUsings.cs`）

```csharp
using System.Runtime.InteropServices;
using Microsoft.Data.Sqlite;
using ActivityMonitor.Core.Models;
```

### 4.4 访问修饰符

- 所有类型和成员必须显式声明访问修饰符（不依赖默认值）
- `private` 是缺省值，但必须写出来

```csharp
// ✅ 正确
private int _counter;
public string Name { get; }

// ❌ 错误
int _counter;      // 默认 private 但省略了修饰符
string Name { get; }
```

### 4.5 var

- 当类型从右侧明显可见时使用 `var`
- 当类型不明确或对可读性有损时显式写出类型

```csharp
// ✅ 正确
var event = new ActivityEvent();
var dict = new Dictionary<string, int>();

// ❌ 不明确，不要用
var result = GetData();              // GetData() 返回什么？
var process = Process.GetProcessById(id);  // 类型不直观
```

### 4.6 空值处理

```csharp
// ✅ 推荐：使用模式匹配
if (windowTitle is null)
{
    return string.Empty;
}

// ✅ 推荐：null 条件运算符
var length = windowTitle?.Length ?? 0;

// ❌ 避免：双重转换
if (obj is string) { var s = (string)obj; }
// ✅ 推荐：模式匹配直接转换
if (obj is string s) { /* 使用 s */ }
```

## 五、注释规范

### 5.1 何时写注释

- **公共 API**（公开方法/属性/类）：写 XML 文档注释 `/// <summary>`
- **复杂算法**：写为什么这么做，而不只是做什么
- **Windows API P/Invoke**：必须标注对应的原生 DLL 和函数签名来源
- **性能敏感代码**：标注为什么采用特定实现

### 5.2 注释风格

```csharp
/// <summary>
/// 获取当前前台窗口的进程信息。
/// </summary>
/// <param name="handle">窗口句柄</param>
/// <returns>进程 ID 和名称；若失败返回 null</returns>
public ProcessInfo? GetForegroundProcess(IntPtr handle)
{
    // Win32 API: GetWindowThreadProcessId 是获取前台窗口进程
    // 最轻量的方式，不可被托管 API 替代（性能路径）
}
```

### 5.3 不要写什么注释

```csharp
// ❌ 无意义注释
int i = 0;  // 设置 i 为 0

// ❌ 注释掉的代码 —— 直接删除，需要时从 git 历史找回
// private int _oldField;
```

## 七、异步与并发规范

### 7.1 异步原则

- I/O 操作（数据库读写、文件操作）使用 `async` / `await`
- CPU 密集型操作不使用 `async`（用 `Task.Run` 仅当需要避免阻塞 UI 线程）
- 追踪轮询（2s 定时器）**不使用 async**，用 `System.Timers.Timer` 回调

### 7.2 线程安全

```csharp
// 追踪引擎的线程模型：
// 1. Timer 回调在 ThreadPool 线程执行
// 2. 窗口状态变更通过锁保护
// 3. DB 写入通过队列串行化，避免多线程并发写

private readonly object _stateLock = new();

public void UpdateState()
{
    lock (_stateLock)
    {
        // 更新当前活动记录
    }
}
```

### 7.3 WPF 线程

- UI 更新（Dashboard、托盘提示）必须调度到 `Dispatcher` 线程
- 后台追踪引擎禁止直接操作 UI 控件

```csharp
// ✅ 正确
Application.Current.Dispatcher.Invoke(() =>
{
    TrayIcon.ToolTipText = $"今日已记录 {hours} 小时";
});
```

## 八、Windows API 调用规范

### 8.1 P/Invoke 声明规则

- 所有 P/Invoke 集中放在 `Win32/NativeMethods.cs`
- 每个声明必须标注 DLL 名称和函数作用
- 使用 `DllImport` + `ExactSpelling` 以提升性能

```csharp
// NativeMethods.cs — Win32 API 声明
internal static partial class NativeMethods
{
    /// <summary>
    /// user32.dll — 获取当前前台窗口句柄
    /// </summary>
    [DllImport("user32.dll", ExactSpelling = true)]
    internal static extern IntPtr GetForegroundWindow();

    /// <summary>
    /// user32.dll — 获取窗口标题长度
    /// </summary>
    [DllImport("user32.dll", CharSet = CharSet.Unicode, ExactSpelling = true)]
    internal static extern int GetWindowText(IntPtr hWnd, StringBuilder text, int count);
}
```

### 8.2 错误处理

```csharp
internal static extern int GetWindowThreadProcessId(IntPtr hWnd, out uint processId);

public static uint? SafeGetProcessId(IntPtr hWnd)
{
    if (hWnd == IntPtr.Zero)
    {
        return null;
    }

    GetWindowThreadProcessId(hWnd, out uint processId);
    return processId;
}
```

## 九、数据库与数据层规范

### 9.1 SQLite 操作

- 所有数据库操作走 Repository 模式，不直接写 SQL 到业务代码
- SQL 语句中使用**参数化查询**，禁止字符串拼接
- 批量写入使用事务包裹

```csharp
// ✅ 正确
public async Task InsertBatchAsync(List<ActivityEvent> events)
{
    using var connection = await _db.GetConnectionAsync();
    using var transaction = connection.BeginTransaction();

    using var cmd = connection.CreateCommand();
    cmd.CommandText = @"
        INSERT INTO activity_events (start_time, category, process_name, window_title)
        VALUES (@start, @cat, @proc, @title)";

    foreach (var e in events)
    {
        cmd.Parameters.Clear();
        cmd.Parameters.AddWithValue("@start", e.StartTime.ToString("O"));
        cmd.Parameters.AddWithValue("@cat", e.Category);
        cmd.Parameters.AddWithValue("@proc", e.ProcessName);
        cmd.Parameters.AddWithValue("@title", e.WindowTitle);
        cmd.ExecuteNonQuery();
    }

    transaction.Commit();
}

// ❌ 禁止
var sql = $"INSERT INTO activity_events VALUES ('{startTime}', ...)";
```

### 9.2 索引策略

```sql
-- 核心查询模式：按天查 + 按类别聚合
CREATE INDEX idx_events_date ON activity_events(start_time);
CREATE INDEX idx_events_cat  ON activity_events(category, start_time);
CREATE INDEX idx_events_proc ON activity_events(process_name, start_time);
```

## 十二、异常处理规范

### 12.1 追踪引擎

- 单个轮询周期的异常不能导致整个引擎崩溃
- 捕获并记录日志后继续下一周期

```csharp
public void OnTimerTick(object? sender, ElapsedEventArgs e)
{
    try
    {
        var hWnd = NativeMethods.GetForegroundWindow();
        // ... 正常逻辑
    }
    catch (Exception ex)
    {
        Logger.Warn(ex, "WindowTracker tick failed, continuing");
        // 不抛出，不崩溃
    }
}
```

### 12.2 自定义异常

- 只定义在确实需要调用方特殊处理的场景
- 继承 `Exception`，后缀 `Exception`

## 十三、性能敏感路径（性能准则）

以下路径标记为性能敏感，需额外关注：

| 路径 | 约束 | 原因 |
|------|------|------|
| `WindowTracker.OnTimerTick` | < 1ms 执行时间 | 每 2 秒执行一次 |
| DB 批量写入 | 每 30 秒一次，< 10ms | 不在轮询周期内写入 |
| `GetWindowText` | 合理 Buffer 大小（256 chars） | 避免重复分配 |
| `ActivityCategorizer.Classify` | 无反射、无 LINQ 高分配 | 每次窗口变化时调用 |

```csharp
// ✅ 性能路径：避免内存分配
[SkipLocalsInit]
private static string SafeGetWindowTitle(IntPtr hWnd)
{
    // 使用栈上 buffer 避免堆分配
    Span<char> buffer = stackalloc char[256];
    unsafe
    {
        fixed (char* p = buffer)
        {
            var len = NativeMethods.GetWindowText(hWnd, new IntPtr(p), buffer.Length);
            return len > 0 ? new string(p, 0, len) : string.Empty;
        }
    }
}
```

## 十五、Agent 开发前必读流程

> 适用于：所有 Claude Code agent（包括 Workflow 中的子 agent）

### 15.1 每次开始编码前

所有 agent **必须在写第一行代码之前**依次执行以下步骤：

```
Step 1  读取 doc/09-会话检查点.md
        → 了解当前进度、中断位置、未完成的任务
        → 如果会话检查点 says "下一步是做 X"，不要跳过

Step 2  读取 doc/07-运行报错日志.md
        → 搜索关键词匹配当前要写的模块（如 "WindowTracker"、"SQLite"）
        → 如果有相关报错记录，避免重复踩坑

Step 3  读取 doc/04-拆分任务.md 中自己的任务清单
        → 明确当前模块的接口依赖和验收标准
        → 确认前置条件是已完成状态

Step 4  检查目标文件是否已存在
        → 如果文件已有代码，在其基础上增量开发
        → 不要覆盖已有实现

Step 5  开始编码
```

### 15.2 开发过程中

| 时机 | 动作 |
|------|------|
| 完成一个最小任务 | 更新 `doc/09-会话检查点.md` 的"任务完成情况" |
| 遇到报错 | 修复后写入 `doc/07-运行报错日志.md` |
| 做了关键决策（A vs B 二选一） | 记入 `doc/09-会话检查点.md` 的"关键决策记录" |
| 发现自己要改的代码和另一角色重叠 | 暂停，先确认职责边界（查 `doc/04-拆分任务.md`） |

### 15.3 会话结束时

如果工作被打断或主动结束，**必须更新** `doc/09-会话检查点.md`：

- 哪些任务完成了（✅）
- 哪些任务做到一半（🔄），当前做到哪一步
- 下一步该做什么
- 有什么 blocker 或遗留问题
- 代码是否已 commit（分支名是什么）

### 15.4 最小可执行任务原则

每个 agent 每次领到的任务应当满足以下条件：

1. **可独立完成** — 不依赖其他角色的中间产物（接口定义等公共契约除外）
2. **可在一次上下文中完成** — 单个 task 不应超过 200 行代码或 3 个文件
3. **有明确验收标准** — 要么测试通过，要么输出产物明确
4. **可回退** — 做坏了不影响已有代码（git 分支或 worktree 隔离）

> **判断标准**：如果 agent 做到一半中断了，下一个 agent 能通过检查点+s 最小任务描述在 5 分钟内接上，而不需要重读整个项目，就算粒度合格。
