## 四、6 个 Agent 角色定义

所有角色定义在 `.claude/agents/` 下，每个 agent 有严格的职责边界和工具限制：

| Agent | 文件名 | 负责模块 | 限制 |
|-------|--------|---------|------|
| 技术架构师 | `architect.md` | Wave 0: M1-M4 | 只做数据模型/接口/DB/Win32 |
| 后端引擎工程师 | `backend-engineer.md` | T1-T5 追踪引擎 | 不做 UI/分类/报表 |
| 内容分析工程师 | `content-analyst.md` | C1-C6 内容解析 | 不做引擎/UI/DB Schema |
| WPF 前端工程师 | `frontend-engineer.md` | U1-U5 界面 | 不做后端逻辑，可 Mock |
| 数据报表工程师 | `report-engineer.md` | R1-R5 报表 | 不做引擎/UI/分类 |

## 六、启动多 agent 开发的两种方式

### 方式 A：单窗口 Workflow（推荐）

在当前会话写一个 Workflow 脚本，所有 agent 在同一个窗口跑：

```js
// 1. Wave 0 串行
await agent('实现 M1-M4', { label: '架构师' })

// 2. Wave 1 并行（每个 agent 独立 worktree）
await parallel([
  () => agent('实现 T1-T5', { label: '引擎', isolation: 'worktree' }),
  () => agent('实现 C1-C6', { label: '内容分析', isolation: 'worktree' }),
  () => agent('实现 U1-U5', { label: '前端', isolation: 'worktree' }),
  () => agent('实现 R1-R5', { label: '报表', isolation: 'worktree' }),
  () => agent('实现 E1-E5', { label: '测试', isolation: 'worktree' }),
])
```

**优点**：一个窗口看全局进度，Workflow 自动处理依赖和合并
**缺点**：上下文仍然会增长（但 Workflow 的子 agent 有自己的上下文，不占用你的主窗口上下文）

### 方式 B：多窗口分别开发

1. 当前窗口做 Wave 0（架构师工作）
2. 完成后，开 **5 个新终端窗口**，每个窗口跑一个角色的工作
3. 每个窗口读这个交接文档恢复上下文
4. 各自在 `feature/engine`、`feature/classification` 等分支开发
5. 最后 PR 合入 main

**优点**：上下文完全独立
**缺点**：需要手动协调分支和合并
